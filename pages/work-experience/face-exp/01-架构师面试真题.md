### Q1: 如何进行微服务拆分？拆分的原则和边界是什么？

**考察点：** 服务拆分方法论、领域驱动设计、实际项目经验

**参考答案：**

微服务拆分应遵循以下原则：

```
┌─────────────────────────────────────────────────────────┐
│                  微服务拆分原则                          │
├─────────────────────────────────────────────────────────┤
│  1. 单一职责：一个服务只做一件事                          │
│  2. 高内聚低耦合：服务内部紧密，服务间松散                │
│  3. 独立部署：每个服务可独立开发、测试、部署              │
│  4. 业务边界：按业务领域拆分，而非技术层                  │
│  5. 数据独立：每个服务有自己的数据库                      │
└─────────────────────────────────────────────────────────┘
```

**拆分策略：**

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| 按业务域拆分 | 用户、商品、订单、支付 | 电商、SaaS |
| 按子域拆分 | DDD限界上下文 | 复杂业务系统 |
| 按团队拆分 | 康威定律 | 大型团队 |

**实际经验（电商项目）：**

```
电商系统拆分示例：
├── 用户服务（user-service）     → 用户注册、登录、会员等级
├── 商品服务（product-service）  → 商品CRUD、SKU管理、库存查询
├── 订单服务（order-service）    → 下单、状态流转、拆单合单
├── 支付服务（payment-service）  → 支付渠道、回调、对账
├── 库存服务（inventory-service）→ 库存扣减、预占、回滚
├── 营销服务（marketing-service）→ 优惠券、活动、积分
└── 搜索服务（search-service）   → ES索引、搜索、推荐
```

**拆分粒度权衡：**
- 太粗：失去微服务意义，部署效率低
- 太细：运维复杂度高，服务间调用链长
- 建议：从粗粒度开始，根据实际需要逐步拆分

### Q6: 如何设计分布式锁？有哪些实现方案？

**考察点：** 分布式锁原理、实现方案对比、实际应用

**参考答案：**

**实现方案对比：**

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| Redis SETNX | 性能高、实现简单 | 需处理超时、主从切换 | 高并发场景 |
| Redisson | 功能完善、可重入 | 依赖Redis | 生产推荐 |
| ZooKeeper | 强一致、可靠性高 | 性能较低 | 强一致性要求 |
| 数据库 | 简单可靠 | 性能差 | 低并发 |

**Redisson 实现：**

```java
// 1. 引入依赖
// implementation 'org.redisson:redisson-spring-boot-starter:3.23.4'

// 2. 使用分布式锁
@Autowired
private RedissonClient redisson;

public void processOrder(String orderId) {
    RLock lock = redisson.getLock("order:lock:" + orderId);
    try {
        // 尝试获取锁，等待3秒，锁定10秒
        boolean acquired = lock.tryLock(3, 10, TimeUnit.SECONDS);
        if (acquired) {
            // 执行业务逻辑
            doProcess(orderId);
        } else {
            throw new BusinessException("获取锁失败");
        }
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    } finally {
        if (lock.isHeldByCurrentThread()) {
            lock.unlock();
        }
    }
}

// 3. 看门狗机制（自动续期）
// Redisson默认启用watchdog，每10秒续期一次
```

**锁的注意事项：**

```
┌────────────────────────────────────────────────────┐
│               分布式锁注意事项                       │
├────────────────────────────────────────────────────┤
│ 1. 设置超时时间，防止死锁                            │
│ 2. 释放锁时验证所有权（Lua脚本原子操作）             │
│ 3. 主从切换可能导致锁丢失（RedLock解决）             │
│ 4. 可重入锁需要记录持有计数                          │
│ 5. 业务执行时间不能超过锁超时时间                    │
└────────────────────────────────────────────────────┘

### Q7: 限流算法有哪些？如何实现分布式限流？

**考察点：** 限流算法原理、实现方案、实际应用

**参考答案：**

**限流算法对比：**

```
┌─────────────────────────────────────────────────────────────┐
│                      限流算法对比                            │
├───────────┬─────────────┬─────────────┬─────────────────────┤
│   算法    │    原理     │    特点     │      适用场景        │
├───────────┼─────────────┼─────────────┼─────────────────────┤
│ 固定窗口   │ 时间窗口内计数│ 实现简单    │ 精度要求不高        │
│ 滑动窗口   │ 细粒度时间窗口│ 精度高      │ 需要精确限流        │
│ 令牌桶     │ 固定速率放令牌│ 允许突发    │ API限流（推荐）     │
│ 漏桶       │ 固定速率流出  │ 严格平滑    │ 流量整形            │
└───────────┴─────────────┴─────────────┴─────────────────────┘
```

**令牌桶算法实现（Redis + Lua）：**

```lua
-- Lua脚本实现令牌桶
local key = KEYS[1]
local capacity = tonumber(ARGV[1])  -- 桶容量
local rate = tonumber(ARGV[2])      -- 放入速率
local requested = tonumber(ARGV[3]) -- 请求数量
local now = tonumber(ARGV[4])       -- 当前时间

local lastRefill = redis.call('hget', key, 'lastRefill')
local tokens = redis.call('hget', key, 'tokens')

if lastRefill == false then
    tokens = capacity
    lastRefill = now
end

-- 计算补充的令牌
local delta = math.max(0, now - lastRefill)
local filledTokens = math.min(capacity, tokens + delta * rate)

if filledTokens >= requested then
    local newTokens = filledTokens - requested
    redis.call('hmset', key, 'tokens', newTokens, 'lastRefill', now)
    return 1  -- 允许
else
    return 0  -- 拒绝
end
```

**考察点：** 分布式事务方案、CAP权衡、实际应用

**参考答案：**

**分布式事务方案对比：**

| 方案 | 一致性 | 性能 | 复杂度 | 适用场景 |
|------|--------|------|--------|----------|
| 2PC | 强一致 | 低 | 高 | 传统系统 |
| TCC | 最终一致 | 中 | 高 | 金融交易 |
| Saga | 最终一致 | 高 | 中 | 长流程业务 |
| 本地消息表 | 最终一致 | 高 | 低 | 简单场景 |
| 事务消息 | 最终一致 | 高 | 低 | 异步场景 |

**TCC实现示例（订单+库存）：**

```java
// TCC事务接口
public interface InventoryTccService {
    
    // Try: 预扣库存
    @TwoPhaseBusinessAction(name = "inventoryDeduct", 
                            commitMethod = "commit", 
                            rollbackMethod = "rollback")
    boolean tryDeduct(@BusinessActionContextParameter(paramName = "orderId") String orderId,
                      @BusinessActionContextParameter(paramName = "skuId") String skuId,
                      @BusinessActionContextParameter(paramName = "quantity") int quantity);
    
    // Confirm: 确认扣减
    boolean commit(BusinessActionContext context);
    
    // Cancel: 回滚预扣
    boolean rollback(BusinessActionContext context);
}

// Try实现
public boolean tryDeduct(String orderId, String skuId, int quantity) {
    // 1. 检查库存是否充足
    // 2. 冻结库存（inventory_frozen表）
    // 3. 记录事务日志
    return inventoryMapper.freezeStock(skuId, quantity, orderId);
}

// Cancel实现
public boolean rollback(BusinessActionContext context) {
    String orderId = context.getActionContext("orderId").toString();
    // 释放冻结库存
    return inventoryMapper.releaseFrozenStock(orderId);
}
```

├─────────────────────────────────────────────────────┤
│  C (Consistency)  : 一致性 - 所有节点数据一致        │
│  A (Availability) : 可用性 - 每个请求都有响应        │
│  P (Partition)    : 分区容错 - 网络分区时系统可用    │
│                                                     │
│  定理：分布式系统中，CAP最多只能同时满足两个         │
└─────────────────────────────────────────────────────┘

         ┌───────────┐
         │     CA     │ ← 单机数据库（MySQL单机）
         └───────────┘
              ╱ ╲
             ╱   ╲
┌───────────┐     ┌───────────┐
│    CP     │     │    AP     │
└───────────┘     └───────────┘
  ZooKeeper         Cassandra
  Redis集群         Eureka
```

**实际架构权衡：**

| 场景 | 选择 | 原因 |
|------|------|------|
| 银行转账 | CP | 一致性优先，可接受短暂不可用 |
| 商品展示 | AP | 可用性优先，短暂不一致可接受 |
| 订单创建 | CP | 库存一致性必须保证 |
| 用户评论 | AP | 可用性优先，最终一致即可 |

**BASE理论（AP的补充）：**

```
BA (Basically Available)  : 基本可用 - 允许部分降级
S (Soft State)            : 软状态 - 允许中间状态
E (Eventually Consistent) : 最终一致 - 一定时间后一致
```

def hybrid_search(query, top_k=10):
    # 1. 语义检索
    semantic_results = vector_db.search(
        embedding_model.encode(query),
        top_k=top_k
    )
    
    # 2. 关键词检索（BM25）
    keyword_results = elastic_search.search(
        query=query,
        top_k=top_k

    merged = reciprocal_rank_fusion(
        semantic_results, 
        keyword_results,
        k=60
    )
    
    # 4. 重排序
    reranked = reranker.rerank(query, merged[:20])
    
    return reranked[:top_k]

### Q13: vLLM的PagedAttention原理是什么？为什么能提升推理性能？

**考察点：** 模型推理优化、显存管理

**参考答案：**

**传统推理 vs PagedAttention：**

```
传统推理（连续内存）：
┌─────────────────────────────────────────────────────┐
│ Request 1: [████████████████████]                   │
│ Request 2: [██████████░░░░░░░░░░]  ← 预分配但未用完 │
│ Request 3: [████████████░░░░░░░░]                   │
│                                                     │
│ 问题：内存碎片、预分配浪费、无法共享                 │
└─────────────────────────────────────────────────────┘

PagedAttention（分页内存）：
┌─────────────────────────────────────────────────────┐
│ Block Pool: [■][■][■][■][■][■][■][■][■][■][■][■]... │
│              ↑     ↑     ↑                          │
│ Request 1: Block 0,1,2,3 (动态分配)                 │
│ Request 2: Block 4,5 (按需分配)                     │
│ Request 3: Block 6,7,8 (共享前缀 Block 0,1)         │
│                                                     │
│ 优势：按需分配、共享前缀、零碎片                     │
└─────────────────────────────────────────────────────┘
```

**核心优化：**

| 优化点 | 说明 | 收益 |
|--------|------|------|
| PagedAttention | KV Cache分块管理 | 显存利用率提升2-4x |
| Continuous Batching | 动态批处理 | 吞吐提升2-3x |
| 前缀缓存 | 共享Prompt前缀 | 延迟降低30-50% |
| Tensor Parallelism | 多卡并行 | 支持更大模型 |

### Q16: 如何评估RAG系统的检索质量？

**考察点：** RAG评估指标、评估方法

**参考答案：**

**RAGAS评估框架：**

| 指标 | 说明 | 计算方式 |
|------|------|----------|
| Faithfulness | 生成答案与检索内容的一致性 | LLM评估 |
| Answer Relevancy | 答案与问题的相关性 | LLM评估 |
| Context Precision | 检索内容的精确度 | 检索内容相关性 |
| Context Recall | 检索内容的召回率 | 与标准答案对比 |

```python
from ragas import evaluate
from ragas.metrics import (
    faithfulness,
    answer_relevancy,
    context_precision,
    context_recall,
)

# 评估数据集
dataset = {
    'question': ['问题1', '问题2'],
    'answer': ['答案1', '答案2'],
    'contexts': [['上下文1'], ['上下文2']],
    'ground_truth': ['标准答案1', '标准答案2'],
}

# 执行评估
result = evaluate(
    dataset,
    metrics=[
        faithfulness,
        answer_relevancy,
        context_precision,
        context_recall,
    ]
)

print(result)
# {'faithfulness': 0.85, 'answer_relevancy': 0.90, ...}
```

**离线评估流程：**

```
1. 构建测试集（问题+标准答案+相关文档）
2. 运行RAG系统生成答案
3. 计算各维度指标
4. 分析低分case，针对性优化
```

**ReAct模式：**

```
ReAct = Reasoning + Acting

┌─────────────────────────────────────────────────────┐

│                                                     │
└─────────────────────────────────────────────────────┘

用户：北京今天天气怎么样？

Thought: 用户想知道北京今天的天气，我需要调用天气查询工具
Action: weather_api(city="北京")
Observation: {"temp": 25, "weather
