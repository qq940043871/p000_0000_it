### 7. 消息队列在系统集成中的作用，如何保证消息可靠性？

**真题来源：** 阿里、字节、美团高频题

**参考答案：**

**消息队列作用：**

| 作用 | 说明 |
|------|------|
| **解耦** | 生产者和消费者不需要直接交互 |
| **异步** | 提升响应速度，非核心流程异步处理 |
| **削峰** | 应对流量洪峰，保护后端服务 |
| **广播** | 一条消息多个消费者消费 |

**可靠性保障：**

| 环节 | 措施 |
|------|------|
| **发送端** | 确认机制（ACK）、重试机制 |
| **存储端** | 持久化、多副本同步 |
| **消费端** | 手动ACK、幂等处理 |
| **整体** | 死信队列、消息追踪 |

**主流MQ对比：**

| 特性 | RabbitMQ | Kafka | RocketMQ |
|------|----------|-------|----------|
| **吞吐量** | 中 | 高 | 高 |
| **延迟** | 微秒级 | 毫秒级 | 毫秒级 |
| **事务** | 支持 | 不支持 | 支持 |
| **顺序消息** | 单队列 | 单分区 | 单队列 |
| **适用场景** | 业务消息 | 日志、大数据 | 金融、交易 |

### 8. 如何设计系统间的数据同步方案？

**真题来源：** 阿里、腾讯企业应用

**参考答案：**

**同步方式对比：**

| 方式 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| **实时同步** | 数据变更立即同步 | 低延迟 | 系统耦合高 |
| **准实时同步** | 定时批量同步 | 性能好 | 有延迟 |
| **异步同步** | 消息队列异步 | 解耦 | 最终一致性 |

**技术方案：**

```
方案1：CDC（Change Data Capture）
数据库 → Canal/Debezium → 消息队列 → 目标系统

方案2：双写
应用 → 主数据库
     → 消息队列 → 从数据库

方案3：ETL
定时任务 → 抽取数据 → 转换 → 加载到目标
```

**注意事项：**
- 数据一致性校验
- 冲突处理策略
- 监控与告警
- 回滚机制

### 13. 如何设计API的认证授权方案？

**真题来源：** 阿里、腾讯安全岗位

**参考答案：**

**认证方案对比：**

| 方案 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| **Session-Cookie** | 服务端存储Session | 简单 | 不适合分布式 |
| **JWT** | 无状态Token | 无状态、跨服务 | 无法主动失效 |
| **OAuth2** | 授权框架 | 标准化、安全 | 复杂 |
| **API Key** | 固定密钥 | 简单 | 安全性低 |

**JWT认证流程：**
```
1. 用户登录 → 验证账号密码
2. 服务端生成JWT → 返回给客户端
3. 客户端存储Token（localStorage/cookie）
4. 后续请求携带Token（Authorization: Bearer xxx）
5. 网关验证Token → 解析用户信息 → 转发请求
```

**JWT结构：**
```
Header.Payload.Signature

Header: {"alg": "HS256", "typ": "JWT"}
Payload: {"userId": 123, "role": "admin", "exp": 1234567890}
Signature: HMACSHA256(base64(Header) + "." + base64(Payload), secret)
```

**安全建议：**
- 使用HTTPS传输
- Token设置合理过期时间
- 敏感信息不要放Payload
- 实现Token刷新机制
- 关键操作二次验证

### 14. 如何处理API版本兼容性问题？

**真题来源：** 美团、京东应用开发

**参考答案：**

**版本管理策略：**

| 策略 | 方式 | 示例 | 优缺点 |
|------|------|------|--------|
| **URL路径** | URL中包含版本 | `/v1/users`, `/v2/users` | 简单直观，但不优雅 |
| **Query参数** | 参数指定版本 | `/users?version=1` | 灵活但不够明确 |
| **Header** | 请求头指定 | `Accept: application/vnd.api.v1+json` | 干净但调试不便 |
| **域名** | 不同域名 | `api1.example.com` | 隔离彻底但成本高 |

**兼容性处理：**
```java
// 版本适配器模式
public interface UserService {
    UserVO getUser(Long id);
}

@Service("userServiceV1")
public class UserServiceV1Impl implements UserService {
    // V1实现
}

@Service("userServiceV2")
public class UserServiceV2Impl implements UserService {
    // V2实现，新字段处理
}

@RestController
public class UserController {
    @Autowired
    @Qualifier("userServiceV1")
    private UserService userServiceV1;
    
    @Autowired
    @Qualifier("userServiceV2")
    private UserService userServiceV2;
    
    @GetMapping({"/v1/users/{id}", "/v2/users/{id}"})
    public UserVO getUser(@PathVariable Long id, HttpServletRequest request) {
        String version = extractVersion(request);
        return "v2".equals(version) ? 
            userServiceV2.getUser(id) : userServiceV1.getUser(id);
    }
}
```

### 15. 如何设计API限流和熔断机制？

**真题来源：** 阿里、字节高频题

**参考答案：**

**限流算法：**

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **固定窗口** | 固定时间窗口计数 | 简单 | 临界突发问题 |
| **滑动窗口** | 窗口滑动计数 | 平滑 | 内存消耗大 |
| **令牌桶** | 恒定速率生成令牌 | 允许突发 | 实现稍复杂 |
| **漏桶** | 恒定速率流出 | 流量整形 | 不允许突发 |

**限流维度：**
- 全局限流：保护系统整体
- IP限流：防止单用户刷接口
- 用户限流：防止单用户滥用
- 接口限流：保护核心接口

**熔断实现（Sentinel）：**
```java
// 定义规则
FlowRule rule = new FlowRule();
rule.setResource("getUser");
rule.setGrade(RuleConstant.FLOW_GRADE_QPS);
rule.setCount(100); // 每秒100次

// 熔断规则
DegradeRule degradeRule = new DegradeRule("getUser");
degradeRule.setGrade(CircuitBreakerStrategy.ERROR_RATIO.getType());
degradeRule.setCount(0.5); // 错误率50%
degradeRule.setTimeWindow(10); // 熔断10秒

// 使用
try (Entry entry = SphU.entry("getUser")) {
    // 业务逻辑
} catch (BlockException e) {
    // 限流或熔断处理
    return "系统繁忙，请稍后重试";
}
```

### 17. 如何处理跨域问题？

**真题来源：** 前端/全栈岗位必考题

**参考答案：**

**跨域原因：**
- 协议不同
- 域名不同
- 端口不同

**解决方案：**

| 方案 | 原理 | 适用场景 |
|------|------|---------|
| **CORS** | 服务端设置响应头 | 主流方案 |
| **代理** | 同源服务器代理 | 开发环境 |
| **JSONP** | 利用script标签 | 仅GET、老系统 |
| **WebSocket** | 不受同源限制 | 实时通信 |

**CORS配置：**
```java
@Configuration
public class CorsConfig implements WebMvcConfigurer {
    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/**")
            .allowedOriginPatterns("*")
            .allowedMethods("GET", "POST", "PUT", "DELETE", "OPTIONS")
```

```java
    response.setContentType("application/vnd.openxmlformats-officedocument.spreadsheetml.sheet");

            new AnalysisEventListener<User>() {

                        success++;
                    } catch (Exception e) {
                        fail++;
                    }
                }

                public void doAfterAllAnalysed(AnalysisContext context) {

                    task.setSuccessCount(success);
                    task.setFailCount(fail);
                    taskService.update(task);
```

```java
    private static final String ALGORITHM = "AES/GCM/NoPadding";
    
    public static String encrypt(String data, String key) throws Exception {
        SecretKeySpec secretKey = new SecretKeySpec(key.getBytes(), "AES");
        Cipher cipher = Cipher.getInstance(ALGORITHM);
        cipher.init(Cipher.ENCRYPT_MODE, secretKey);
        byte[] encrypted = cipher.doFinal(data.getBytes());
        return Base64.getEncoder().encodeToString(encrypted);
    }

        SecretKeySpec secretKey = new SecretKeySpec(key.getBytes(), "AES");
        Cipher cipher = Cipher.getInstance(ALGORITHM);
        cipher.init(Cipher.DECRYPT_MODE, secretKey);
        byte[] decrypted = cipher.doFinal(Base64.getDecoder().decode(encryptedData));
        return new String(decrypted);
    }
}
```

### 37. 如何实现API签名验证？

**真题来源：** 开放平台、第三方对接

**参考答案：**

**签名流程：**
```
1. 拼接参数（按key排序）
2. 加入timestamp和nonce
3. 使用密钥进行HMAC签名
4. 请求携带签名
5. 服务端验证签名和时间戳
```

**实现代码：**
```java
// 签名生成
public class SignatureUtil {
    
    public static String generate(Map<String, String> params, String secretKey) {
        // 1. 参数排序
        String sortedParams = params.entrySet().stream()
            .sorted(Map.Entry.comparingByKey())
            .map(e -> e.getKey() + "=" + e.getValue())
            .collect(Collectors.joining("&"));
        
        // 2. 加入时间戳和随机数
        long timestamp = System.currentTimeMillis();
        String nonce = UUID.randomUUID().toString();
        String content = sortedParams + "&timestamp=" + timestamp + "&nonce=" + nonce;
        
        // 3. HMAC-SHA256签名
        Mac mac = Mac.getInstance("HmacSHA256");
        mac.init(new SecretKeySpec(secretKey.getBytes(), "HmacSHA256"));
        String signature = Hex.encodeHexString(mac.doFinal(content.getBytes()));
        
        return signature;
    }
    
    public static boolean verify(Map<String, String> params, String signature, 
                                  String secretKey, long maxAge) {
        // 验证时间戳
        long timestamp = Long.parseLong(params.get("timestamp"));
        if (System.currentTimeMillis() - timestamp > maxAge) {
            return false;
        }
        
        // 验证签名
        String expectedSignature = generate(params, secretKey);
        return expectedSignature.equals(signature);
    }
}
```

### 40. 缓存策略与常见问题解决方案？

**真题来源：** 各大厂高频题

**参考答案：**

**缓存策略选择：**

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| **Cache-Aside** | 先查缓存，不存在查DB | 读多写少 |
| **Write-Through** | 写缓存同时写DB | 数据一致性要求高 |
| **Write-Behind** | 先写缓存，异步写DB | 写性能优先 |

**三大问题解决：**

| 问题 | 解决方案 |
|------|---------|
| **缓存穿透** | 布隆过滤器、缓存空对象 |
| **缓存击穿** | 互斥锁、热点数据永不过期 |
| **缓存雪崩** | 随机过期时间、多级缓存 |

**分布式锁解决击穿：**
```java
public Object getWithLock(String key) {
    // 1. 查询缓存
    Object value = redis.get(key);
    if (value != null) {
        return value;
    }
    
    // 2. 获取分布式锁
    String lockKey = "lock:" + key;
    boolean locked = redis.setnx(lockKey, "1", 10);
    
    if (locked) {
        try {
            // 3. 再次检查缓存（Double Check）
            value = redis.get(key);
            if (value != null) {
                return value;
            }
            
            // 4. 查询数据库
            value = db.query(key);
            
            // 5. 写入缓存
            if (value != null) {
                redis.set(key, value, 3600);
            } else {
                // 缓存空对象，短过期
                redis.set(key, "NULL", 60);
            }
            
            return value;
        } finally {
            redis.del(lockKey);
        }
    } else {
        // 获取锁失败，等待后重试
        Thread.sleep(50);
        return getWithLock(key);
    }
}
```

### 49. 如何处理技术债务？

**真题来源：** 技术负责人岗位

**参考答案：**

**技术债务类型：**
| 类型 | 说明 |
|------|------|
| **代码债务** | 代码质量差、重复代码 |
| **架构债务** | 设计不合理、耦合高 |
| **文档债务** | 文档缺失、过时 |
| **测试债务** | 测试覆盖不足 |
| **依赖债务** | 依赖版本过旧 |

**处理策略：**
1. **识别记录**：记录技术债务清单
2. **评估影响**：影响范围、修复成本
3. **优先级排序**：根据风险和收益
4. **纳入迭代**：随新需求逐步偿还
5. **预防机制**：Code Review、技术评审

### 50. 描述一次你解决复杂技术问题的经历（STAR法则）

**真题来源：** 各大厂必考题

**参考答案框架：**

| 要素 | 内容 |
|------|------|
| **S（情境）** | 项目背景、遇到的问题 |
| **T（任务）** | 目标是什么、需要解决什么 |
| **A（行动）** | 具体做了什么、技术方案、协调工作 |
| **R（结果）** | 量化成果、学到了什么 |

**示例：**
> **S**：在电商平台项目中，秒杀场景下系统经常出现超卖问题。
>
> **T**：需要确保在高并发情况下库存扣减的准确性，避免超卖。
>
> **A**：
> 1. 分析问题根因：数据库行锁在高并发下成为瓶颈
> 2. 设计方案：Redis Lua脚本预扣库存 + MQ异步创建订单
> 3. 实现幂等性：用户ID + 商品ID作为去重Key
> 4. 添加监控：库存预警、订单异常告警
>
> **R**：
> - QPS从1000提升到50000
> - 超卖率降为0
> - 订单处理延迟控制在100ms以内
