   - 降级策略：推理失败返回缓存或提示

**追问回答：**

- **性能优化**：量化(INT4/INT8)、Tensor Parallel、Flash Attention
- **成本控制**：小模型+RAG、缓存复用、按需扩缩容

```python
# 混合检索实现
def hybrid_search(query, k=10):
    # 1. 语义检索
    semantic_results = vector_db.search(
        embedding(query), 
        top_k=k*2
    )
    
    # 2. 关键词检索 (BM25)
    keyword_results = elasticsearch.search(
        query,
        top_k=k*2
    )
    
    # 3. 融合排序 (RRF)
    merged = reciprocal_rank_fusion(
        semantic_results, 
        keyword_results
    )
    
    # 4. 重排
    reranked = reranker.rerank(
        query, 
        merged[:k*2]
    )
    
    return reranked[:k]
```

### Q7: 向量检索原理

**标准答案：**

**HNSW (Hierarchical Navigable Small World) 算法：**

```
原理:
- 构建多层图结构
- 上层稀疏(快速跳跃)、下层稠密(精确搜索)
- 搜索时从上层入口点开始，逐层向下找最近邻

图结构示意:

Layer 2:     ●───────────────●        (稀疏)
             │               │
Layer 1:     ●───────●───────●        (中等)
             │       │       │
Layer 0:     ●───●───●───●───●        (稠密)
             ↑
          查询点

搜索过程:
1. 从Layer 2入口点开始
2. 在当前层找最近邻
3. 进入下一层继续搜索
4. 直到Layer 0返回结果
```

**关键参数：**

| 参数 | 说明 | 影响 |
|------|------|------|
| M | 每层最大连接数 | 越大召回率越高、索引越大 |
| efConstruction | 构建时搜索范围 | 越大索引质量越好、构建越慢 |
| efSearch | 搜索时搜索范围 | 越大召回率越高、搜索越慢 |

**性能指标：**

```
召回率: Recall@K = 命中真实Top-K / K
QPS: 每秒查询数
延迟: P50 / P99

典型配置:
- M = 16
- efConstruction = 200
- efSearch = 100

性能参考 (Qdrant, 100万向量):
- Recall@10: 98%+
- QPS: 5000+
- P99延迟: <50ms
```

### Q10: LoRA 微调原理

**标准答案：**

**原理详解：**

```
原始模型权重: W (冻结)

LoRA添加: ΔW = B × A
- A: (d × r) 降维矩阵
- B: (r × d) 升维矩阵
- r: 秩 (通常 r << d)

最终权重: W' = W + B × A

参数量对比:
- 原始: d × d
- LoRA: 2 × d × r
- 当 r=8, d=4096: 参数量减少 99.6%
```

**训练过程：**

```python
from peft import LoraConfig, get_peft_model
from transformers import AutoModelForCausalLM

# 加载基础模型
model = AutoModelForCausalLM.from_pretrained(
    "Qwen/Qwen2.5-7B-Instruct",
```

### Q12: ROI 评估方法

**标准答案：**

**成本计算：**

```
成本 = 硬件成本 + 推理成本 + 人力成本

硬件成本:
├── GPU服务器: 4×A100 80G ≈ 40万/年(租赁)
├── 存储: 10TB SSD ≈ 2万/年
└── 网络: 内网带宽 ≈ 1万/年

推理成本:
├── API模式: Token单价 × 日均Token数
└── 私有部署: 电费 + 运维人力

人力成本:
├── 开发: 2人 × 30万/年 = 60万
└── 运维: 1人 × 20万/年 = 20万
```

**收益计算：**

```
收益 = 效率提升 + 转化提升 + 成本节约

智能客服案例:
├── 原人工客服: 20人 × 15万/年 = 300万
├── AI客服替代: 60%问题自动解决
├── 节约人力: 12人 × 15万 = 180万
└── 扣除AI成本: 180万 - 60万 = 120万/年

知识库问答案例:
├── 原查询效率: 30分钟/次
├── AI后效率: 3分钟/次
├── 效率提升: 10倍
├── 日均查询: 1000次
└── 年节约时间: 1000 × 27分钟 × 250天 = 112500小时
```

**ROI 公式：**

```
ROI = (收益 - 成本) / 成本 × 100%

案例计算:
- 年收益: 180万
- 年成本: 60万(开发) + 40万(硬件) + 20万(运维) = 120万
- ROI = (180 - 120) / 120 × 100% = 50%

考虑长期:
- 第2年起开发成本下降
- ROI可提升至 100%+
```

### 题目二：RAG 系统优化

**标准答案：**

**问题分析：**

```
准确率60%的可能原因:
├── 分块不合理 (块太大/太小)
├── Embedding模型效果差
├── 检索策略单一 (仅语义检索)
├── Rerank缺失
├── Query理解不足
└── 知识库质量问题 (噪声/重复)
```

**优化方案：**

```
优化1: 分块策略
├── 改用语义分块
├── 块大小: 512-1024 tokens
└── 重叠: 100 tokens
预期提升: 5-10%

优化2: 混合检索
├── 语义检索 + BM25
├── RRF融合排序
└── 预期提升: 10-15%

优化3: Rerank
├── 添加BGE-Reranker
└── 预期提升: 5-10%

优化4: Query优化
├── Query改写/扩展
├── 意图识别
└── 预期提升: 5-10%

总预期提升: 25-45%
目标: 60% → 85%+
```
