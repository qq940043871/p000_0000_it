### 1.1 基础概念

```
Q1: HashMap的底层实现原理是什么？JDK1.8之后有哪些优化？
```

<details>

**参考答案：**

**底层结构**：JDK1.7使用数组+链表，JDK1.8使用数组+链表+红黑树

**优化点：**
1. **数据结构变化**：当链表长度超过8且数组长度≥64时，链表转为红黑树（O(log n)优化）
2. **插入方式**：JDK1.7头插法→JDK1.8尾插法（避免环形链表）
3. **扩容机制**：首次插入16，负载因子0.75，扩容翻倍
4. **计算index**：`tab[i = (n - 1) & hash]`，而非`hash % n`

**关键问题回答：**
```java
// 为什么不用红黑树而用链表？
// - 树的节点是链表节点的2倍，空间开销大
// - 阈值8是统计学结论：泊松分布计算，链表长度≥8概率<0.0000006

// 为什么数组容量是2的幂次？
// - 便于用位运算替代取模，提升性能
// - (n-1) & hash 等价于 hash % n，但更快
```

</details>

```
Q5: 抽象类和接口有什么区别？JDK8之后有什么变化？
```

<details>

**参考答案：**

**JDK8前：**
- 抽象类：单继承，有构造方法，可有普通成员变量
- 接口：多实现，无构造方法，只有常量

**JDK8后：**
- 接口可定义**默认方法**（default）：`default void method() {}`
- 接口可定义**静态方法**：`static void staticMethod() {}`
- JDK9：私有方法

**选择建议：**
- **需要共享代码/状态**→抽象类
- **需要多继承/行为契约**→接口
- **Java API设计**：Comparable、Runnable等用接口

</details>

```
Q6: ArrayList和LinkedList的区别？如何选择？
```

<details>

**参考答案：**

| 操作 | ArrayList | LinkedList |
|------|-----------|------------|
| 随机访问 | O(1) | O(n) |

- 大数据量时内存连续性影响GC

</details>

```
Q8: 详细说说你对代理模式的理解？
```

<details>

**参考答案：**

**静态代理：**
```java
public class UserServiceProxy implements UserService {
    private UserService target;
    
    public UserServiceProxy(UserService target) {
        this.target = target;
    }
    
    @Override
    public void save() {
        System.out.println("前置增强");
        target.save();
        System.out.println("后置增强");
    }
}
```

**动态代理（Spring AOP原理）：**
```java
// JDK动态代理 - 目标类必须实现接口
UserService proxy = (UserService) Proxy.newProxyInstance(
    classLoader,
    target.getClass().getInterfaces(),
    (proxy, method, args) -> {
        System.out.println("前置");
        Object result = method.invoke(target, args);
        System.out.println("后置");
        return result;
    }
);

// CGLIB代理 - 目标类可以没有接口
Enhancer enhancer = new Enhancer();
enhancer.setSuperclass(UserService.class);
enhancer.setCallback(MethodInterceptor);
UserService proxy = (UserService) enhancer.create();
```

**区别：**
- JDK代理：基于接口，性能稍差，Spring默认
- CGLIB：基于继承，性能更好，无法代理final类/final方法

</details>

```
Q10: 什么情况下会触发Full GC？Full GC和Minor GC的区别？
```

<details>

**参考答案：**

**Minor GC（Young GC）：**
- 触发条件：Eden区满
- 频率：较频繁
- 停顿时间：短（通常几十毫秒）
- 回收对象：Eden + Survivor

**Full GC（Major GC）：**
- 触发条件：
  1. 老年代空间不足
  2. Metaspace空间不足
  3. System.gc()（建议JVM执行）
  4. 分配担保失败（大对象直接进老年代）
- 频率：相对较少
- 停顿时间：长（几百毫秒到几秒）

**触发场景详解：**
```java
// 场景1：老年代空间不足
List<byte[]> list = new ArrayList<>();
// 不断向list添加大对象，超过Eden直接进入老年代

// 场景2：分配担保失败
// Minor GC前检查老年代可用空间 < 新生代对象总空间
// 此时会触发Full GC或直接Minor GC+Full GC

// 场景3：Metaspace满
// 大量动态类加载（如CGLIB、ASM生成类）
```

</details>

### 2.2 垃圾回收

```
Q11: 常见的垃圾回收算法有哪些？各有什么优缺点？
```

<details>

**参考答案：**

| 算法 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| 引用计数 | 对象被引用+1，引用消失-1 | 实现简单 | 无法处理循环引用 | 几乎不用 |

**分代收集理论：**
- 新生代：复制算法（对象存活率低）
- 老年代：标记-清除/整理（对象存活率高）

</details>

### 3.2 并发工具

```
Q17: CountDownLatch、CyclicBarrier、Semaphore有什么区别？
```

<details>

**参考答案：**

| 工具 | 作用 | 特点 | 场景 |
|------|------|------|------|
| CountDownLatch | 等待一组操作完成 | 计数只能减，不能重置 | 等待多个线程完成 |
| CyclicBarrier | 等待所有线程到达某点 | 计数可重置，cyclic | 多线程汇合点 |
| Semaphore | 控制并发数量 | 信号量，可acquire/release | 限流 |

**示例：**

```java
// CountDownLatch - 主线程等待子线程
CountDownLatch latch = new CountDownLatch(3);
new Thread(() -> { /* 任务1 */ latch.countDown(); }).start();
new Thread(() -> { /* 任务2 */ latch.countDown(); }).start();
new Thread(() -> { /* 任务3 */ latch.countDown(); }).start();
latch.await(); // 主线程等待
System.out.println("所有任务完成");

// CyclicBarrier - 所有线程汇合后一起继续
CyclicBarrier barrier = new CyclicBarrier(3, () -> System.out.println("全员到齐"));
// 3个线程都调用await()后一起继续

// Semaphore - 限流
Semaphore semaphore = new Semaphore(5);
for (int i = 0; i < 10; i++) {
    new Thread(() -> {
        semaphore.acquire();
        try {
            // 同时只有5个线程执行
        } finally {
            semaphore.release();
        }
    }).start();
}
```

</details>

```
Q18: ConcurrentHashMap是如何保证线程安全的？JDK1.7和1.8有什么区别？
```

<details>

**参考答案：**

**JDK1.7：Segment分段锁**
```java
// 每个Segment类似一个HashMap，锁住一段
Segment<K,V>[] segments;
static class Segment<K,V> extends ReentrantLock {}
// 默认16个Segment，并发度16
```

**JDK1.8：CAS + Synchronized**
```java
// 1.8移除Segment，使用Node数组
// 2. 读操作无锁（volatile）
// 3. 写操作：CAS失败则synchronized锁住单个Node
// 4. 链表过长转红黑树（8→红黑树）
```

**关键改进：**
- 减小锁粒度，提高并发度
- 移除分段锁，降低复杂度
- 使用CAS+CAS，降低竞争

<details>

**参考答案：**

**死锁的必要条件（必须同时满足）：**
1. 互斥：资源只能被一个线程持有

```bash
"Thread-1":

  locked 0x00007f...

# 3. 分析代码中的锁获取顺序
```

**解决死锁：**
```java
// 方法1：统一锁获取顺序（最有效）
// 所有线程都先获取lockA，再获取lockB

// 方法2：使用tryLock带超时
if (lockA.tryLock(1, TimeUnit.SECONDS)) {
    try {
        if (lockB.tryLock(1, TimeUnit.SECONDS)) {
            // 业务逻辑
        }
    } finally {
        lockB.unlock();
        lockA.unlock();
    }
}

// 方法3：减少锁粒度
// 使用分段锁或ConcurrentHashMap
```

```
Q27: MySQL索引底层结构是什么？B+树相比B树有什么优势？
```

<details>

**B+树 vs B树：**

```
B树（每个节点都存数据）:
        [15]
       /    \
   [5,10]    [20,25]

B+树（只有叶子节点存数据）:
        [15]
       /    \
   [5,10]    [20,25]
    /  \       /  \
  ...  ...   ...  ...
  (叶子节点用链表连接)
```

**B+树优势：**

| 特性 | B树 | B+树 |
|------|-----|------|
| 查询稳定性 | 不稳定（深度不定） | 稳定（所有查询到叶子） |
| 范围查询 | 需要回溯 | 链表遍历 |
| 磁盘IO | 多 | 少（矮胖） |
| 内存占用 | 高 | 低（非叶子不加载） |

**InnoDB索引：**
- 主键索引：聚簇索引，B+树，叶子存完整行数据
- 辅助索引：B+树，叶子存主键值

```
Q28: 什么情况下索引会失效？如何优化？
```

<details>

**参考答案：**

**索引失效场景：**

```sql
-- 1. 最左前缀原则
CREATE INDEX idx_name_age ON user(name, age);
-- 失效情况
SELECT * FROM user WHERE age = 20;  -- ✗ 不使用name
SELECT * FROM user WHERE name = '张三' AND age > 20;  -- ✓

-- 2. 类型转换
SELECT * FROM user WHERE name = 123;  -- ✗ 字符串索引遇到数字
SELECT * FROM user WHERE name = '123';  -- ✓

-- 3. 函数/运算
SELECT * FROM user WHERE YEAR(create_time) = 2024;  -- ✗
SELECT * FROM user WHERE create_time >= '2024-01-01';  -- ✓

-- 4. LIKE前缀
SELECT * FROM user WHERE name LIKE '%张%';  -- ✗
SELECT * FROM user WHERE name LIKE '张%';  -- ✓

-- 5. OR条件
SELECT * FROM user WHERE name = '张三' OR age = 20;  -- ✗
-- 改用 UNION 或 给age也建索引

-- 6. IS NULL / IS NOT NULL
SELECT * FROM user WHERE name IS NULL;  -- ✗（除非是组合索引的一部分）
```

**优化建议：**
- 遵循最左前缀原则
- 避免在索引列上使用函数
- 使用覆盖索引减少回表
- 定期分析表 `ANALYZE TABLE`

</details>

```
Q29: MySQL事务隔离级别有哪些？分别解决什么问题？
```

<details>

**参考答案：**

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|---------|------|-----------|------|
| Read Uncommitted | ✓ | ✓ | ✓ |
| Read Committed | ✗ | ✓ | ✓ |
| Repeatable Read (默认) | ✗ | ✗ | ✓ |
| Serializable | ✗ | ✗ | ✗ |

**问题解释：**
- **脏读**：读到其他事务未提交的数据
- **不可重复读**：同一事务两次读取结果不一致
- **幻读**：同一事务两次查询发现多了/少了记录

**MVCC原理（RR级别）：**

```sql
-- Read Committed：每次SELECT都生成新的read_view
-- Repeatable Read：事务开始时生成read_view，整个事务用同一个
```

**InnoDB MVCC实现：**
- 每行有隐藏列：DB_TRX_ID, DB_ROLL_PTR
- 读取数据时：根据read_view判断可见性
- Undo log链：存储历史版本

</details>

```
Q35: ES写入数据的过程是什么？为什么ES能实现近实时搜索？
```

<details>

**参考答案：**

**写入流程：**

```
写入请求 → Routing计算分片 → Primary Shard写入
                        ↓
                Memory Buffer（内存缓冲）
                        ↓
            Refresh（默认1秒）→ Segment（可搜索）
                        ↓
                Translog（保证持久性）
                        ↓
         Flush（分段合并，Buffer清空，Translog清空）
```

**近实时原理：**
1. 写入先到内存Buffer，不立即持久化
2. Refresh每秒执行一次，Buffer→Segment，可被搜索
3. Translog保证数据不丢失
4. 最终Flush到磁盘

**优化参数：**
```yaml
index.refresh_interval: 1s  # 默认1秒刷新
index.translog.durability: async  # 异步刷Translog，性能更高但可能丢数据
```

### 6.1 Kafka

```
Q36: Kafka为什么这么快？它的分区和副本机制是怎样的？
```

<details>

**参考答案：**

**Kafka快的原理：**

| 优化点 | 说明 |
|--------|------|
| 顺序写磁盘 | 追加写入，磁盘顺序IO极快 |
| 零拷贝 | sendfile系统调用，无需CPU拷贝 |
| 批量处理 | 消息积攒批量发送/消费 |
| 压缩 | 支持GZIP/Snappy/LZ4压缩 |
| 分区并行 | 多分区并行消费 |
| 页缓存 | 利用OS页缓存，不占JVM堆 |

**顺序写原理：**
```
传统随机写：磁头不断移动 → 慢
Kafka追加写：始终在文件末尾 → 极快（类似SSD）
```

**分区与副本：**
```
Topic: orders (3分区, 副本因子:3)
├── Partition 0: Leader=Broker-1, ISR=[1,2,3]
├── Partition 1: Leader=Broker-2, ISR=[1,2,3]
└── Partition 2: Leader=Broker-3, ISR=[1,2,3]
```

**副本同步机制：**
- **AR** (Assigned Replicas)：所有副本
- **ISR** (In-Sync Replicas)：同步中的副本（leader + 跟上进度的follower）
- Follower从Leader拉取消息，保持同步

</details>

**幂等生产者原理：**

- 每条消息有SequenceNumber
- Broker根据PID+SeqNum去重

### 6.2 RabbitMQ

```
Q38: RabbitMQ有哪些交换机类型？它们有什么区别？
```

<details>

**参考答案：**

| 交换机类型 | 路由规则 | 典型场景 |
|------------|----------|----------|
| Direct | 完全匹配routing key | 点对点 |
| Fanout | 广播到所有队列 | 广播消息 |
| Topic | routing key模式匹配 | 主题订阅 |
| Headers | 匹配header属性 | 复杂路由 |

**工作原理：**

```java
// Direct: routing key = binding key
channel.exchangeDeclare("direct.exchange", "direct");
channel.queueDeclare("queue1", false, false, false, null);
channel.queueBind("queue1", "direct.exchange", "order.created");

// Fanout: 忽略routing key，广播所有
channel.exchangeDeclare("fanout.exchange", "fanout");
channel.queueBind("queue1", "fanout.exchange", "");  // binding key无意义
channel.queueBind("queue2", "fanout.exchange", "");

// Topic: 支持通配符
// * 匹配一个词
// # 匹配零个或多个词
channel.queueBind("queue1", "topic.exchange", "order.*");    // order.created
channel.queueBind("queue2", "topic.exchange", "order.#");     // order.created.paid
```

</details>
