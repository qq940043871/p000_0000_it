# 03 - 可扩展性设计

> 如何让系统随业务增长而平滑扩展

---

## 📖 概述

可扩展性（Scalability）决定了系统能承载多大的业务增长。本章介绍水平扩展和垂直扩展的策略。

---

## 🏗️ 一、扩展策略

### 1.1 水平扩展 vs 垂直扩展

| 类型 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| **垂直扩展** | 提升单机配置 | 改动小、成本低 | 有物理上限 | 早期、业务稳定 |
| **水平扩展** | 增加机器数量 | 无理论上限 | 架构复杂 | 中后期、高增长 |

### 1.2 扩展维度

```
应用层：┌────┐ ┌────┐ ┌────┐ ┌────┐  →  无限扩容
        │实例1│ │实例2│ │实例3│ │实例4│ 
        └────┘ └────┘ └────┘ └────┘

缓存层：┌────┐ ┌────┐ ┌────┐           →  Redis Cluster
        │节点1│ │节点2│ │节点3│ 
        └────┘ └────┘ └────┘

数据层：┌────┐ ┌────┐ ┌────┐           →  分库分表
        │库1 │ │库2 │ │库3 │ 
        └────┘ └────┘ └────┘
```

### 2.2 弹性伸缩策略

```java
public class ElasticScalingStrategy {
    
    // 基于QPS的扩容
    public ScalingPlan calculateByQPS(int currentQPS, int targetQPS) {
        int currentPods = currentQPS / 100; // 单Pod处理100QPS
        int targetPods = targetQPS / 100;
        
        return ScalingPlan.builder()
            .desiredPods(targetPods)
            .scaleUpThreshold(80)  // CPU>80%扩容
            .scaleDownThreshold(20) // CPU<20%缩容
            .build();
    }
    
    // 基于队列深度的扩容
    public ScalingPlan calculateByQueueDepth(int queueDepth) {
        if (queueDepth > 10000) {
            return ScalingPlan.builder().desiredPods(50).build();
        } else if (queueDepth > 5000) {
            return ScalingPlan.builder().desiredPods(20).build();
        }
        return ScalingPlan.builder().desiredPods(5).build();
    }
}
```

### 3.2 分片策略

```yaml
spring:
  shardingsphere:
    rules:
      sharding:
        tables:
          t_order:
            actualDataNodes: ds_${0..3}.t_order_${0..15}
            tableStrategy:
              standard:
                shardingColumn: order_id
                shardingAlgorithmName: order_inline
            keyGenerateStrategy:
              column: order_id
              keyGeneratorName: snowflake
        sharding-algorithms:
          order_inline:
            type: INLINE
            props:
              algorithm-expression: t_order_${order_id % 16}
```

### 3.3 分片算法

| 算法 | 原理 | 适用场景 |
|------|------|----------|
| **Hash分片** | hash(key) % 分片数 | 均匀分布 |
| **范围分片** | 按ID区间分片 | 时间序列 |
| **一致性Hash** | 虚拟节点环 | 动态扩容 |
| **复合分片** | 多字段组合 | 复杂查询 |

## 📊 四、容量规划

### 4.1 容量评估模型

```python
# 容量评估公式
def calculate_capacity():
    # 输入参数
    daily_orders = 1000000  # 日订单量
    peak_qps = daily_orders * peak_factor / day_seconds
    avg_order_amount = 100  # 平均订单金额
    
    # 单机能力
    single_machine_qps = 500  # 单机处理QPS
    single_machine_concurrent = 1000  # 单机并发
    
    # 扩容计算
    required_machines = ceil(peak_qps / single_machine_qps)
    
    # 冗余考虑
    final_machines = required_machines * redundancy_factor(1.2)
    
    return {
        'daily_orders': daily_orders,
        'peak_qps': peak_qps,
        'required_machines': required_machines,
        'final_machines': final_machines
    }
```

### 4.2 性能基准测试

| 场景 | 单机TPS | 说明 |

| 混合读写 | 2000+ | 含DB操作 |

### 5.1 分布式事务

// Seata AT模式
@GlobalTransactional(name = "createOrder", timeoutMills = 30000)
public Order createOrder(OrderDTO dto) {
    // 1. 创建订单
    Order order = orderDao.create(dto);
    
    // 2. 扣减库存（自动补偿）
    inventoryClient.deduct(dto.getSkuId(), dto.getCount());

    accountClient.deduct(dto.getUserId(), dto.getAmount());
    
    return order;
}

### 5.2 跨分片查询

```java
// 跨分片聚合查询
public List<OrderSummary> getOrderSummaryByDate(LocalDate date) {
    List<Order> allOrders = new ArrayList<>();
    
    // 并行查询所有分片
    for (int i = 0; i < SHARD_COUNT; i++) {
        List<Order> orders = orderDao.queryByDate(date, i);
        allOrders.addAll(orders);
    }
    
    // 内存聚合
    return allOrders.stream()
        .collect(Collectors.groupingBy(Order::getUserId))
        .entrySet().stream()
        .map(e -> new OrderSummary(e.getKey(), e.getValue()))
        .collect(Collectors.toList());
}
```

## 📝 六、最佳实践

1. **无状态化设计**：所有状态外置到Redis/DB
2. **服务自治**：每个服务可独立部署和扩展
3. **渐进式扩展**：先垂直后水平
4. **监控容量**：提前预警，避免被动扩容
5. **灰度验证**：新扩节点先接入少量流量
