# 01 - 系统架构设计

> 大型电商平台架构从0到1的设计思路

---

## 📖 概述

电商系统的架构设计决定了系统的上限。一个好的架构需要满足：**高可用、高性能、可扩展、易维护、成本可控**。

---

## 🏗️ 一、架构演进历程

### 1.1 单体架构（初创期）

```
┌─────────────────────────────────┐
│          单体应用               │
│  ┌─────┬─────┬─────┬─────┐   │
│  │用户 │商品 │订单 │支付 │   │
│  │模块 │模块 │模块 │模块 │   │
│  └─────┴─────┴─────┴─────┘   │
│  ┌─────────────────────────┐   │
│  │      MySQL 数据库       │   │
│  └─────────────────────────┘   │
└─────────────────────────────────┘
```

**特点**：
- 优点：简单、快速迭代、部署简单
- 缺点：扩展性差、单点故障、代码耦合

**适用场景**：用户量 < 10万，日均订单 < 1万

---

### 1.2 垂直拆分架构（成长期）

```
┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│ 用户服务  │  │ 商品服务  │  │ 订单服务  │  │ 支付服务  │
│  :8081   │  │  :8082   │  │  :8083   │  │  :8084   │
└────┬─────┘  └────┬─────┘  └────┬─────┘  └────┬─────┘
     │            │            │            │
┌────▼────┐  ┌───▼────┐  ┌───▼────┐  ┌───▼────┐
│用户库    │  │商品库   │  │订单库   │  │支付库   │
└─────────┘  └────────┘  └────────┘  └────────┘
```

**特点**：
- 按业务域拆分，团队独立
- 数据库按业务拆分，减少跨库join
- 扩展性提升

**适用场景**：用户量 10-100万，日均订单 1-10万

## 🔧 二、核心架构设计原则

### 2.1 高可用设计原则

| 原则 | 说明 | 示例 |
|------|------|------|
| **冗余设计** | 无单点故障 | 多副本部署、主备切换 |
| **降级设计** | 非核心功能可降级 | 促销时关闭评论功能 |
| **熔断设计** | 防止故障蔓延 | Sentinel/Hystrix |
| **超时设计** | 避免请求堆积 | 接口超时设置 |

### 2.2 高性能设计原则

| 原则 | 说明 | 示例 |
|------|------|------|
| **异步化** | 非核心链路异步 | MQ解耦 |
| **缓存化** | 读多写少场景 | 多级缓存 |
| **批量化** | 减少网络开销 | 批量接口 |
| **预计算** | 提前准备热点数据 | 活动预热 |

### 2.3 可扩展设计原则

| 原则 | 说明 | 示例 |
|------|------|------|
| **服务拆分** | 业务边界清晰 | 用户、商品、订单 |
| **数据分片** | 水平扩展数据 | 分库分表 |
| **配置外置** | 减少代码改动 | Nacos配置中心 |
| **插件化** | 功能模块可插拔 | 策略模式 |

## 📊 五、容量评估方法

### 5.1 核心指标计算

```
日PV = 日均UV × 平均页面数
峰值QPS = 日均QPS × 峰值系数 (通常3-10倍)
所需服务器数 = 峰值QPS ÷ 单机QPS
```

### 5.2 性能基准参考

| 场景 | 单机QPS参考 | 说明 |
|------|-------------|------|
| 静态接口 | 5000+ | Nginx直接响应 |
| Redis缓存命中 | 3000+ | 纯内存操作 |
| DB简单查询 | 500-1000 | 走索引 |
| 复杂业务接口 | 100-300 | 含业务逻辑 |

### 5.3 扩容评估公式

```
扩容目标 = 预期QPS ÷ 当前单机QPS × 冗余系数(1.2-1.5)
```

## ⚠️ 六、常见问题与解决方案

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 数据库连接耗尽 | 连接池配置不当 | 合理配置连接池 |
| 服务雪崩 | 单点故障蔓延 | 熔断、降级 |
| 数据不一致 | 分布式事务失败 | TCC/Saga补偿 |
| 接口超时 | 慢查询 | SQL优化、索引 |
| 内存溢出 | 缓存数据过大 | 合理设置过期时间 |

### 7.1 阿里

- **中台战略**：业务中台+数据中台
- **技术栈**：Java+Dubbo+DRDS+OceanBase

- **架构**：JSF微服务框架

- **特点**：高并发、分布式事务

### 7.3 拼多多

- **架构**：弱化SOA，强调服务化
- **技术栈**：Go+Python+Tair
- **特点**：快速迭代、成本控制

### 7.4 美团

- **架构**：OCTO服务框架
- **技术栈**：C++/Java+HBase
- **特点**：高性能、强一致
