# 02 - 高可用架构

> 如何保证电商系统7x24小时稳定运行

---

## 📖 概述

高可用（High Availability）是电商系统的生命线。一次宕机可能造成数百万的损失。本章介绍大厂如何构建高可用架构。

---

## 🏗️ 一、高可用架构设计

### 1.1 什么是高可用

**可用性指标**：
```
可用性 = MTBF / (MTBF + MTTR) × 100%

MTBF (Mean Time Between Failures)：平均故障间隔时间
MTTR (Mean Time To Repair)：平均修复时间
```

| 可用性 | 年故障时间 | 适用场景 |
|--------|-----------|----------|
| 99% | 3.65天 | 普通系统 |
| 99.9% | 8.76小时 | 企业级 |
| 99.99% | 52分钟 | 电商核心 |
| 99.999% | 5分钟 | 金融级 |

### 1.2 高可用架构原则

```
1. 消除单点故障 (Single Point of Failure)
2. 冗余设计 (Redundancy)
3. 故障隔离 (Fault Isolation)
4. 快速故障转移 (Failover)
5. 优雅降级 (Graceful Degradation)
```

## 🔄 三、故障转移机制

### 3.1 应用层故障转移

```java
@Service
public class ServiceDiscovery {
    
    @Autowired
    private LoadBalancer loadBalancer;
    
    // 服务调用 - 自动故障转移
    public Response callService(String serviceName, Request request) {
        List<Instance> instances = serviceRegistry.getHealthyInstances(serviceName);
        
        for (Instance instance : instances) {
            try {
                return httpClient.post(instance.getUrl(), request);
            } catch (Exception e) {
                // 标记为不健康
                serviceRegistry.markUnhealthy(instance);
                log.warn("服务 {} 调用失败，尝试下一个", instance.getUrl());
            }
        }
        
        throw new ServiceUnavailableException("无可用服务实例");
    }
}
```

### 3.2 数据库故障转移

```
┌─────────┐      自动切换       ┌─────────┐
│  主库   │ ─────────────────► │  从库   │
│(读写)   │                    │(只读)   │
└─────────┘                    └─────────┘
     ▲                             │
     │     自动选主                 │
     └─────────────────────────────┘
```

**MySQL MGR 自动故障转移**：
- 多数派投票选主
- 自动脑裂检测
- 故障秒级切换

### 3.3 缓存故障转移

```
正常情况：
┌─────────┐  写入   ┌─────────┐
│ 应用    │ ──────►│ Redis主 │
└─────────┘        └─────────┘

故障时自动切换：
┌─────────┐  写入   ┌─────────┐
│ 应用    │ ──────►│ Redis从 │
└─────────┘        └─────────┘
```

**Sentinel 故障检测**：
```yaml
# Sentinel 配置
sentinel monitor mymaster 127.0.0.1 6379 2
sentinel down-after-milliseconds mymaster 5000
sentinel failover-timeout mymaster 30000
```

## 🔥 四、熔断与降级

### 4.1 熔断器模式

```
                    熔断器状态机
                         
    ┌──────────┐   失败次数超限    ┌──────────┐
    │  关闭     │ ───────────────►│  打开     │
    │ (Closed) │                  │ (Open)   │
    │ 正常熔断  │                  │ 拒绝请求  │
    └────┬─────┘                  └────┬─────┘
         │                              │
         │  成功                         │  半开
         │  ◄────────────────────────  │
         │                              │
         │                        ┌─────▼─────┐
         │                        │   半开     │
         │                        │ (Half-Open)│
         │                        │  探测请求  │
         └────────────────────────┴───────────┘
```

**Sentinel 实现**：
```java
@SentinelResource(value = "getProduct", 
                  blockHandler = "getProductBlock")
public Product getProduct(Long id) {
    return productService.getById(id);
}

// 熔断降级处理
public Product getProductBlock(Long id, BlockException e) {
    // 返回降级数据
    return Product.defaultValue();
}
```

### 4.2 降级策略

| 策略 | 说明 | 示例 |
|------|------|------|
| **接口降级** | 非核心接口关闭 | 关闭商品推荐 |
| **功能降级** | 简化业务流程 | 下单不验库存 |
| **页面降级** | 返回静态页面 | 活动页静态化 |
| **返回兜底** | 返回默认值 | 返回空列表 |

### 4.3 限流策略

```java
// 基于Sentinel的限流
@SentinelResource(value = "createOrder",
                  blockHandler = "createOrderBlock")
public Order createOrder(OrderDTO dto) {
    return orderService.create(dto);
}

public Order createOrderBlock(OrderDTO dto, BlockException e) {
    throw new BusinessException("系统繁忙，请稍后再试");
}
```

**限流算法**：
- 计数器算法
- 滑动窗口算法
- 漏桶算法
- 令牌桶算法
