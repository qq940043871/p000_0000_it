# 08 - 数据一致性与分布式事务

> 如何在分布式环境下保证数据一致性

---

## 📖 概述

分布式事务是电商系统的核心难点。本章介绍CAP、BASE理论与常见的分布式事务解决方案。

---

## 🏗️ 一、CAP 定理

### 1.1 理论

```
                CAP 定理
                    │
        ┌───────────┼───────────┐
        │           │           │
        ▼           ▼           ▼
   ┌────────┐  ┌────────┐  ┌────────┐
   │   C   │  │   A   │  │   P   │
   │一致性  │  │可用性  │  │分区容错│
   └────────┘  └────────┘  └────────┘
   
   结论：只能同时满足 CA、CP 或 AP
   原因：网络分区不可避免，必须在 C 和 A 之间权衡
```

### 1.2 电商场景选择

| 场景 | CAP选择 | 说明 |
|------|---------|------|
| 支付核心 | CP | 强一致性优先 |
| 商品查询 | AP | 可用性优先 |
| 订单创建 | CP | 必须一致 |

---

## 🔄 二、BASE 理论

### 2.1 核心思想

```
BASE = Basically Available + Soft State + Eventually Consistent

基本可用：系统在故障时允许降级
软状态：允许中间状态存在
最终一致：经过一段时间后达到一致
```

### 2.2 vs 传统ACID

| 特性 | ACID | BASE |
|------|------|------|
| 一致性 | 强一致 | 最终一致 |
| 可用性 | 低 | 高 |
| 扩展性 | 差 | 好 |
| 适用场景 | 金融、订单 | 电商、社交 |

## 🛠️ 三、分布式事务解决方案

### 3.1 Seata AT 模式（推荐）

```java
@GlobalTransactional(name = "createOrder", timeoutMills = 30000)
public Order createOrder(OrderDTO dto) {
    // 1. 创建订单
    Order order = orderDao.insert(dto);
    
    // 2. 扣减库存（自动补偿）
    inventoryClient.deduct(dto.getSkuId(), dto.getCount());
    
    // 3. 扣减余额
    accountClient.deduct(dto.getUserId(), dto.getAmount());
    
    // 4. 发送消息
    mqProducer.send("order-topic", order);
    
    return order;
}
```

**AT模式原理**：
```
1. TM（事务管理器）开启全局事务
2. TM向TC（事务协调器）注册分支事务
3. 各分支执行SQL，生成undo_log
4. 提交时TC协调各分支提交/回滚
5. 异常时根据undo_log自动回滚
```

### 3.2 TCC 模式

```java
@Service
public class InventoryTccService {
    
    @Autowired
    private InventoryDao inventoryDao;
    
    // Try: 预留资源
    @Transactional
    public boolean tryDeduct(Long skuId, Integer count) {
        // 冻结库存
        return inventoryDao.freezeStock(skuId, count) > 0;
    }
    
    // Confirm: 确认扣减
    @Transactional
    public boolean confirmDeduct(Long skuId, Integer count) {
        // 真正扣减库存，删除冻结记录
        return inventoryDao.realDeduct(skuId, count) > 0;
    }
    
    // Cancel: 取消预留
    @Transactional
    public boolean cancelDeduct(Long skuId, Integer count) {
        // 释放冻结的库存
        return inventoryDao.unfreezeStock(skuId, count) > 0;
    }
}
```

### 3.3 Saga 模式

```
Saga执行流程：

订单创建 ──► 库存扣减 ──► 支付扣款 ──► 订单完成
    │            │            │
    ▼            ▼            ▼
  成功       成功        失败/补偿
    │            │            │
    └────────────┴────────────┘
                 │
                 ▼
           补偿事务执行
           退款 ──► 库存恢复 ──► 订单取消
```

### 3.4 本地消息表

```java
@Service
public class OrderService {
    
    @Autowired
    private OrderDao orderDao;
    
    @Autowired
    private MessageDao messageDao;
    
    @Transactional
    public void createOrder(OrderDTO dto) {
        // 1. 创建订单
        orderDao.insert(dto);
        
        // 2. 记录消息（本地消息表）
        messageDao.insert(new Message()
            .setBizType("ORDER_CREATE")
            .setBizId(dto.getOrderId())
            .setStatus(0)  // 0=待发送
            .setRetryCount(0));
    }
}

// 消息发送定时任务
@Scheduled(fixedDelay = 1000)
public void sendPendingMessages() {
    List<Message> messages = messageDao.selectPending(100);
    
    for (Message msg : messages) {
        try {
            // 发送消息
            mqProducer.send(msg.getTopic(), msg.getContent());
            
            // 更新状态
            messageDao.updateStatus(msg.getId(), 1);
        } catch (Exception e) {
            // 失败重试
            messageDao.incrementRetry(msg.getId());
        }
    }
}
```

## ⚠️ 四、方案对比

| 方案 | 一致性 | 复杂度 | 性能 | 适用场景 |
|------|--------|--------|------|----------|
| **2PC/3PC** | 强一致 | 高 | 低 | 不推荐 |
| **Seata AT** | 强一致 | 低 | 中 | 通用场景 |
| **TCC** | 最终一致 | 高 | 高 | 高并发场景 |
| **Saga** | 最终一致 | 中 | 高 | 长流程 |
| **本地消息表** | 最终一致 | 中 | 高 | 非核心场景 |
| **MQ事务消息** | 最终一致 | 中 | 高 | 异步解耦 |

## 🎯 五、实战建议

### 5.1 选择原则

| 场景 | 推荐方案 |
|------|----------|
| 同库多表事务 | 本地事务 |
| 跨库强一致 | Seata AT |
| 高并发核心链路 | TCC |
| 超长业务流程 | Saga |
| 非核心异步 | MQ消息 |

### 5.2 最佳实践

```java
// 1. 事务边界要小
@Transactional  // 控制在方法级别
public void createOrder() {
    // 事务内只做核心操作
}

// 2. 避免长事务
@GlobalTransactional(timeoutMills = 5000)  // 设置超时
public void createOrder() {
    // 快速执行，快速释放资源
}

// 3. 补偿机制要幂等
@Retryable(value = Exception.class, maxAttempts = 3)
public void compensate(Order order) {
    // 补偿操作必须幂等
}
```
